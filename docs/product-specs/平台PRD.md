## 快速摘要

- 平台一期要做的是一条可运营、可审核、可通知的数字藏品闭环，而不是交易平台。
- 核心主链路是“发码 -> 领藏品 -> 编辑内容 -> 审核 -> 展示”，评论、转让、消息围绕这条主链路展开。
- 后台重点是发行、治理、审核、客服与权限；C 端重点是领取、编辑、公开展示、互动和转让。
- 一期优先交付稳定规则边界，不追求复杂编辑器、复杂交易和深度经营分析。

## 一期目标

- 建立从发行到公开展示的完整业务闭环。
- 让后台可以稳定管理系列、批次、激活码、内容审核、评论审核、转让记录和会员。
- 让会员可以完成登录、激活、编辑、提交审核、公开展示、评论和转让。
- 让关键状态变更都有权限控制、审核记录和通知反馈。

## 一期不做

- 复杂交易市场与撮合机制
- 深度富文本或模块化内容编辑器
- 点赞、收藏、举报等完整互动体系
- 深度经营分析与自动化运营编排

## 关键角色

- 后台角色：超级管理员、发行运营、内容审核员、评论审核员、会员运营或客服。
- C 端角色：未登录访客、登录会员、藏品拥有者。
- 角色差异的详细口径不在本页展开，统一看 [用户角色与权限视角](./用户角色与权限视角.md)。

## 关键业务对象

- 系列：藏品的大类目，是批次的归属对象。
- 发行批次：一次具体发行行为，承载数量、有效期和批次状态。
- 激活码：会员领取藏品的唯一凭证；生成激活码时同步生成对应藏品。
- 藏品：数字资产本体，激活前待领取，激活后归属到会员。
- 藏品内容版本：会员编辑后的展示内容实体，支持草稿、审核、公开、下架与版本替换。
- 会员：统一承接小程序和公众号登录的 C 端账户。
- 评论：围绕公开藏品产生的互动内容，支持审核和二级回复。
- 转让单：一笔藏品转让流程实例，承载方式、状态、双方与有效期。
- 通知消息：系统对会员发送的站内或站外业务消息。

## 核心主链路

1. 后台创建系列和发行批次，并生成激活码。
2. 会员登录后输入激活码，完成藏品领取。
3. 会员编辑藏品展示内容，保存草稿并提交审核。
4. 系统完成机审与人工审核，审核通过后更新公开展示内容。
5. 访客可访问公开页，登录会员可评论；运营侧可继续治理评论与异常内容。
6. 满足条件时，拥有者可发起转让；系统负责状态切换、归属变更和通知。

## 能力范围

### 后台能力

- 仪表盘：提供激活、审核、评论、转让等关键总览和待办提醒。
- 发行管理：管理系列、批次、激活码，以及导出、作废、发放记录。
- 藏品与内容治理：查看藏品、当前公开内容、待审版本、审核历史，并执行冻结、解冻、下架、复核。
- 评论治理：查看评论上下文，执行审核、屏蔽、删除等治理动作。
- 会员与客服：查看会员基础信息、渠道绑定、名下藏品、内容、评论与风险记录。
- 转让治理：查看转让记录、转让状态与异常单据，必要时取消未完成转让。
- 通知中心：管理模板、站内信、站外发送记录与失败原因。
- 系统管理：管理后台用户、角色、菜单、字典与日志。

### C 端能力

- 登录与账户归并：支持微信小程序和公众号授权，并按统一会员口径归并账户。
- 激活藏品：输入激活码并领取藏品。
- 我的藏品：查看名下藏品状态，进入编辑、转让和记录查询。
- 内容编辑：维护标题、封面、简介等展示内容，支持保存草稿、提交审核、查看驳回原因和清空编辑内容。
- 公开展示与评论：公开页展示基础信息和内容，登录会员可评论，拥有者可回到编辑入口。
- 转让：支持指定会员转让和转让码转让，并查看或接收转让结果。
- 消息中心：查看未读或已读消息并跳转业务详情。

## 核心业务约束

- 后台用户体系与会员体系严格隔离，不共用权限模型。
- 激活码生成与藏品创建绑定，不能只发码不建藏品。
- 新版本未审核通过前，旧公开版本继续在线。
- 有草稿或待审核内容的藏品不能发起转让。
- 同一藏品同一时间只能存在一个有效转让单。
- 转让成功后，藏品归属权和编辑权同时转移，但公开页地址保持稳定。
- 审核、下架、转让、激活等关键事件都应保留记录，并触发消息通知能力。

## 一期验收口径

- 能完整跑通“发码 -> 激活 -> 编辑 -> 审核 -> 展示 -> 评论 -> 转让 -> 通知”。
- 后台和 C 端都能按角色边界工作，且关键操作受权限控制。
- 激活码、藏品、内容、评论、转让、会员等状态定义一致，能与设计和实现对齐。
- 审核记录、操作日志、通知记录可以支撑后续治理和排查。

## 后续预留

- 更强的内容编辑器与展示模板
- 更完整的互动能力，如点赞、收藏、举报
- 更复杂的通知编排和多渠道重试
- 更细颗粒度的风控、黑名单和异常行为识别
- 更完整的数据分析和经营看板
