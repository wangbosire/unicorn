# 二期-通知与转让运营增强

## 文档目标

明确二期范围、里程碑、基建选型和完成标准。该计划替代《二期准备与文档收口计划》成为当前唯一 active 计划。

## 当前状态

- 状态：`active`
- 阶段定位：一期 `V1` 已闭环并归档，二期聚焦"通知编排"与"转让运营"两块运营增强
- 启动基线：截至 `2026-05-19`，文档收口工作已完成；本计划正式立项

## 快速摘要

- 二期不引入新业务功能（如点赞、收藏、互动），只把一期已交付能力的运营深度补齐。
- 异步基建首次引入：`BullMQ + Redis` 承载通知派发与可重试副作用。
- 按四个主题里程碑推进，优先级从用户感知出发：先派发基建与会员撤销，再做运营治理。
- 完成后通知派发具备重试与可观测，转让具备撤销、补偿和审计闭环。

## 范围

### 范围内

- 通知派发的异步化、重试、死信、可观测
- 后台通知模板管理（CRUD、变量占位、多渠道差异化文案、启停）
- 会员端转让撤销（接收方确认前可撤回）
- 转让异常态补偿与回滚的运营入口
- 转让运营人工处置留痕和审计字段补齐
- `BullMQ + Redis` 基建接入（含 Docker Compose、健康检查、配置约定）

### 范围外

- 互动能力扩展（点赞、收藏、举报）
- 内容编辑器与展示模板深化
- 跨端重构、跨包架构调整
- 第三方消息渠道新增（仅完善现有的站内信 / 订阅消息 / 公众号）

## 里程碑

里程碑命名按主题词，不再延用 `M` 序号。

### 里程碑一：派发基建

- 引入 `BullMQ + Redis`：新增 `docker-compose` 服务、运行时配置、健康检查
- 抽出通知派发服务，把现有同步发送切换到队列消费
- 支持基础重试策略（指数退避）与死信队列
- 实现会员端转让撤销链路（API + 小程序入口 + 状态机更新）
- 完成标准：通知派发完全异步、可重试；会员可在接收方未确认时撤回转让
- 状态：`done`（2026-05-19）
  - 派发器：`apps/api/src/modules/notifications/notification-dispatcher.service.ts`
  - Worker：`apps/api/src/modules/notifications/notification.processor.ts`
  - 撤销 API：`POST /member-api/my/transfers/:transferId/cancel`
  - 接入点：激活、内容审核通过/驳回/下架、评论审核通过/驳回/屏蔽、转让创建/完成/过期/撤销
  - 回归脚本：`pnpm test:phase2-dispatch`

### 里程碑二：模板治理

- 后台通知模板管理页（列表、详情、新增、编辑、停用）
- 模板支持变量占位与多渠道差异化文案（站内信 / 订阅消息 / 公众号）
- 模板版本与启停开关；派发时按当前生效版本渲染
- 替换现有硬编码文案为模板引用
- 完成标准：所有一期通知文案可在后台维护，不再依赖代码发版

### 里程碑三：派发可观测

- 派发记录扩展：每条通知的渠道明细、重试历史、最终结果与失败原因
- 后台派发明细页与重试/重投操作
- 失败聚合视图（按模板 / 渠道 / 原因维度）
- 完成标准：运营可定位单条派发失败并手动重投

### 里程碑四：转让运营

- 转让异常态识别：超时未释放、已扣未达等
- 后台运营处置入口（强制完成、强制回滚、补偿记录）
- 处置动作留痕与审计字段补齐
- 完成标准：客诉和运营事故的转让兜底通路存在并可追溯

### 观察性专项（里程碑外，搭车里程碑一一并落地）

- 自定义 `JsonLogger`，所有日志输出为单行结构化 JSON
- `RequestIdMiddleware` + `AsyncLocalStorage` 贯通 HTTP 请求生命周期；响应回写 `x-request-id`
- 21 个 service 全量接入 Nest `Logger`，状态变更、关键动作、业务拒绝点结构化打点
- `ApiExceptionFilter` 自动携带 `requestId`、`code`、HTTP 元信息
- `GET /api/public-api/health?deep=1` 探活 MySQL + Redis（各 500ms 超时）
- 拆分 `acceptTransferByResolver` 为 `ensureTransferAcceptable / expireTransferAndNotify / applyTransferAcceptance / notifyOnTransferAcceptance`
- 状态：`done`（2026-05-19）
  - 真相源：`apps/api/src/common/observability/`
  - 约定：[日志与追踪约定](../../design-docs/engineering/日志与追踪约定.md)
  - 回归脚本：`pnpm test:phase2-observability`

## 基建口径

- **队列**：`BullMQ`（Node 生态成熟、自带 Dashboard、支持延迟队列与重试策略）
- **依赖中间件**：`Redis`（项目当前未接入，本期新增；通过 `docker-compose` 引入）
- **演进方向**：先承载通知派发；如后续需要异步审核、定时任务，可复用同一基建
- **可观测**：队列层用 Bull Dashboard 或后台自建明细页；业务层用派发记录表

## 完成标准

- 通知派发链路全部经过队列；可重试、有死信、有运营明细
- 通知文案完全数据化，无硬编码
- 会员端转让撤销在小程序与 API 都可用
- 转让运营异常态有处置入口且全留痕
- `BullMQ + Redis` 在本地、CI、生产配置三个环境都跑通

## 风险与对策

- 风险：引入 Redis 增加部署复杂度。
  对策：先在 `docker-compose` 中固化，部署文档同步更新；不引入 Redis 集群，单实例起步。

- 风险：队列异步化后原有同步副作用的边界容易模糊。
  对策：里程碑一就明确"主业务流程不依赖队列结果"的口径，通知失败不回滚业务。

- 风险：模板化后历史通知数据与新模板不对齐。
  对策：模板治理上线时只影响新派发，历史派发记录保留原文案快照。

- 风险：转让补偿动作直接改库可能误操作。
  对策：所有运营处置走专用 API，且强制留痕；不开放裸 SQL 入口。

## 决策日志

- 2026-05-19：确认二期范围聚焦"通知 + 转让"两块运营增强，做透不走 MVP。
- 2026-05-19：基建选型 `BullMQ + Redis`，Redis 为新增依赖。
- 2026-05-19：里程碑顺序按用户感知优先级排：派发基建 → 模板治理 → 派发可观测 → 转让运营。
- 2026-05-19：原 active 计划《二期准备与文档收口计划》随本计划立项而归档。
- 2026-05-19：发现一期实际未实现任何通知写入，里程碑一从"迁移同步派发"调整为"首次实现派发"，直接采用异步形态。
- 2026-05-19：里程碑一渠道范围确定：`IN_APP` 真实实现；`MINIAPP_SUBSCRIPTION` 与 `WECHAT_MP` 提供桩适配器，真实下游推迟到后续里程碑。
- 2026-05-19：派发模型确定一条 `NotificationMessage` + 每渠道一条 `BullMQ` 任务，每次尝试写一条 `NotificationDispatchRecord`，保留重试历史。
- 2026-05-19：会员端转让撤销限制为「发起方在 `PENDING_ACCEPT` 状态下」，并通知接收方；转让码模式（无指定接收方）撤销时跳过通知。
- 2026-05-19：通知类型枚举新增 `TRANSFER_CANCELLED` 与 `TRANSFER_EXPIRED`，对齐状态机。
- 2026-05-19：观察性专项搭车里程碑一一并落地——结构化 JSON 日志、请求级 trace-id、健康深度探活、关键 service 重构均已完成。
- 2026-05-19：日志输出格式选定 JSON 而非 Nest 默认人类可读形式，原因是为后续日志聚合（ELK / Loki / Datadog）零成本铺路；实现上用 Nest `LoggerService` 接口写自定义实现，不引入 pino，保持依赖最小。
- 2026-05-19：trace-id 在 BullMQ Worker 上下文不可用（不同事件循环），约定 Worker 日志改用 `jobId` 替代 `requestId`，并在文档中固化。
