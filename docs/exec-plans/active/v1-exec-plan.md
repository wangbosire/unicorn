# V1 执行计划

## 文档目标

基于现有 PRD，将一期需求拆分为研发可执行的模块优先级、依赖关系与开发顺序，作为设计、排期与实施基线。

## 当前状态

- 状态：`active`
- 阶段定位：一期业务主链路准备落地阶段的总执行计划
- 最近关注点：M1 已通过验收（见 [M1 验收清单](../completed/m1-acceptance-checklist.md)）；优先收敛 **M2 内容展示闭环** 的设计、实现与验收口径

## 目标结果

- 形成一期研发范围的统一优先级划分
- 明确模块依赖关系与推荐推进顺序
- 为各阶段里程碑提供统一验收口径
- 为后续专项计划、开发实现与测试验收提供上位基线

## 范围内事项

- 一期业务目标拆分与优先级分层
- 关键业务域之间的依赖关系
- 推荐开发顺序与阶段交付结果
- M1 到 M4 的阶段性里程碑与验收口径

## 范围外事项

- 每个模块的详细接口字段设计
- 前端页面交互稿与视觉细节
- 基础设施部署方案与发布流水线细节
- 二期及以后版本的长期路线图

## 排期原则

- 优先打通主链路：“发码 -> 领藏品 -> 编辑内容 -> 审核 -> 展示”。
- 优先交付高依赖基础模块，再交付运营增强模块。
- 先完成最小可用闭环，再补评论、转让、通知等扩展链路。
- 审核、转让、通知等复杂模块均依赖统一账户、藏品模型与状态模型稳定后再推进。

## 依赖关系

### 基础前置依赖

- 会员体系是 C 端全部能力前置依赖。
- RBAC、后台用户、菜单管理是全部后台页面前置依赖。
- 系列/发行批次是激活码生成前置依赖。
- 激活码生成是藏品生成前置依赖。
- 藏品主模型是内容版本、审核、转让的前置依赖。
- 内容版本模型是公开展示、人工复核、评论挂载的前置依赖。

### 关键链路依赖

- 评论体系依赖公开展示页上线。
- 转让体系依赖藏品归属模型、内容版本状态模型稳定。
- 通知中心依赖会员体系、审核事件、转让事件稳定。
- 仪表盘依赖各业务模块先产生稳定统计口径。

## 风险与对策

- 风险：会员体系、RBAC、藏品模型等底座能力如果迟迟不稳定，会阻塞多个业务域并行推进
- 对策：优先固化共享枚举、核心实体与权限边界，避免在前端和业务模块中重复发散

- 风险：内容审核、评论审核、转让、通知等扩展链路过早启动，会分散主链路闭环资源
- 对策：严格按 P0 -> P1 -> P2 顺序推进，以 M1 和 M2 作为第一阶段完成标准

- 风险：里程碑如果只写范围不写验收口径，后续容易出现“做了很多但无法判断是否完成”
- 对策：所有阶段成果都应绑定明确的演示脚本、状态检查和测试覆盖要求

## 推荐开发顺序

### 阶段 1：底座建设

- 会员体系
- 后台用户、角色、菜单、RBAC
- 系列管理
- 发行批次管理

交付结果：

- 后台可登录并完成权限控制。
- C 端会员可通过微信渠道完成统一登录。
- 后台可维护系列和批次。

### 阶段 2：发行与资产生成

- 激活码管理基础能力
- 生成激活码时同步生成藏品
- 藏品主模型与状态模型
- 激活流程

交付结果：

- 后台可生成激活码与待领取藏品。
- 会员可通过激活码领取藏品。

### 阶段 3：内容编辑与公开

- 藏品内容版本模型
- 内容编辑页基础能力
- 草稿与待审核版本
- 公开展示页

交付结果：

- 会员可编辑自己的藏品内容。
- 平台已有最小可访问的公开展示页。

### 阶段 4：审核闭环

- 内容机审
- 人工复核列表
- 审核记录
- 下架兜底页

交付结果：

- 藏品内容审核闭环打通。
- 平台具备基础风险治理能力。

### 阶段 5：评论与会员运营

- 评论模型
- 评论审核
- 会员管理
- 登录日志、操作日志、字典管理

交付结果：

- 公开页具备互动能力。
- 后台具备会员与评论治理能力。

### 阶段 6：转让与通知

- 转让单模型
- 指定会员转让
- 转让码转让
- 消息中心
- 站外通知

交付结果：

- 藏品支持流转。
- 关键业务事件支持站内外通知。

### 阶段 7：仪表盘与收尾

- 仪表盘核心指标
- 激活码导出与作废
- 运营视图完善

交付结果：

- 运营、审核、客服可通过首页快速处理待办与查看总体状态。
