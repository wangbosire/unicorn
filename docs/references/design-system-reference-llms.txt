面向 LLM 的设计系统参考笔记

用途
- 以紧凑、可复用、贴近仓库语境的形式沉淀设计系统相关结论，方便人和智能体引用。

当前仓库背景
- 后台管理端基于 React + Vite + shadcn/ui。
- 后台当前以 `shadcn-admin` 作为 UI 基座，而不是把它当作完整产品模板直接照搬。
- 现有代码应尽量延续已建立的视觉语言，而不是随意引入无关风格。

仓库内结论
- 后台页面应继承现有 `shadcn-admin` 基座中的布局、导航、响应式和可访问性模式。
- 新业务页面应该像“在现有后台产品上扩展出的页面”，而不是彼此孤立的演示页。
- 共享 UI 组件只应在多个页面模式稳定后再抽离，不要过早把一次性业务组件塞进共享包。

组件与页面指导
- 优先沿用现有 dashboard 体系中的侧边栏、顶栏、卡片、筛选区、表格、详情页和表单布局模式。
- 列表页默认采用“筛选区 + 表格 + 行操作 + 详情或编辑流”的结构，而不是重新发明交互外壳。
- 后台状态展示优先使用明确的状态 badge、时间戳和操作人/操作记录。
- 对冻结、作废、驳回、撤销等高风险操作，应提供明确确认、必要的原因输入和可审计痕迹。

当前已观察到的后台页面模式
- 页面头部通常由“面包屑式模块路径 + 页面标题 + 一句话说明 + 右上角主操作按钮”组成。
- 列表页常见结构是：页头 -> 可选筛选卡片 -> 数据列表卡片。
- 筛选区目前以 `Card` 承载，内部常使用 `Input` 等基础控件，以简洁的单行或网格布局组织。
- 数据列表通常以 `Card` 包裹 `Table`，卡片头部显示列表标题和数据总量说明。
- 表格首列常使用更高强调度的文本，如编号、名称、主键或关键识别字段。
- 状态字段优先使用 `Badge` 展示，并通过 `default` / `secondary` 等变体区分可用、停用、待处理等状态。
- 时间字段倾向格式化为本地可读时间，而不是直接展示原始时间戳。
- 列表空态、加载态、错误态统一在表格内部以整行提示呈现，而不是跳出列表容器单独展示。

建议采用的交互模式
- “新增”“批量生成”“导出”等主操作优先放在页面头部右上角，使用一到两个最核心按钮承载。
- “筛选条件”适合单独放在卡片中，与表格本体分离，避免表格头部拥挤。
- 详情查看、编辑、局部变更等操作可以优先考虑表格行操作进入详情页、抽屉或编辑流。
- 破坏性或不可逆动作应优先通过确认弹窗承接，而不是直接在行内立即执行。

确认与风险操作模式
- 当前仓库已有通用 `ConfirmDialog` 组件，可用于删除、停用、作废、驳回、冻结等风险动作。
- 确认弹窗适合承载标题、说明、附加提示和确认按钮状态，不建议每个业务页各自重新实现。
- 当动作依赖表单提交时，可通过 `form` 方式让确认按钮直接提交对应表单。

状态 badge 映射规范
- `Badge` 的 `default` 变体优先用于“启用 / 正常 / 已完成 / 已领取 / 已公开 / 已通过”这类正向或稳定状态。
- `Badge` 的 `secondary` 变体优先用于“待处理 / 未发放 / 待领取 / 冻结 / 停用 / 待接收”这类中性或待办状态。
- `Badge` 的 `destructive` 变体建议保留给“已作废 / 已驳回 / 已下架 / 高风险异常”这类需要明显风险提示的状态；当前业务页里还未系统化落地，但后续可按此扩展。
- `Badge` 的 `outline` 变体更适合弱强调标签、分类标签或非主状态信息，不建议拿来承载核心业务状态主表达。
- 同一类状态在不同页面中应保持同一语义方向，例如“启用 / 正常 / 已完成”不要有的用 `default`、有的又退回 `secondary`。
- 状态展示文案应优先先做“后端枚举 -> 中文标签”的显式转换，再传给 UI，而不是把后端原值直接散落在页面上。

后台列表页字段优先级规范
- 第 1 优先级字段：唯一识别字段，如编号、主键、编码、模板标识、转让单号、激活码。这类字段通常放首列，并使用更高强调度。
- 第 2 优先级字段：业务主信息，如名称、系列、批次、事件名、会员昵称、角色名。这类字段用于帮助快速理解当前记录是什么。
- 第 3 优先级字段：归属与关系字段，如所属系列、归属会员、来源渠道、转入方/转出方、关联藏品。
- 第 4 优先级字段：状态字段，如启用/停用、审核状态、领取状态、转让状态，建议稳定放在表格中后段但仍保持可快速扫描。
- 第 5 优先级字段：时间字段，如创建时间、失效时间、处理时间、激活时间，通常靠后展示。
- 第 6 优先级字段：长文本字段，如描述、备注、摘要，默认后置，并结合 `truncate` 或详情页承接，避免挤压核心识别信息。

后台列表页排序建议
- 默认列顺序通常遵循“识别字段 -> 主信息 -> 关系字段 -> 状态 -> 时间 -> 备注/描述”。
- 如果页面最核心目标是处理待办而不是识别记录，可将“状态”适度前移，但不建议压过唯一识别字段。
- 若页面存在多个状态维度，例如藏品状态、内容状态、审核状态，应尽量连续排布，方便横向对比。

不建议做的事
- 不要把第三方展示页直接当成产品级信息架构。
- 不要引入和当前后台基座无关的视觉母题。
- 不要由前端页面自行定义业务状态含义，状态语义应以产品和后端文档为准。

使用建议
- 设计新后台页面或组件时，优先扩展既有的布局、间距、表单和表格模式。
- 一旦共享 UI 决策趋于稳定，应从零散页面实现沉淀到共享包或设计文档。
- 第三方模板只能作为基座和参考，不应作为产品设计系统的唯一真相源。

建议后续补充的内容
- `apps/admin` 当前实际使用的间距尺度与字体选择
- 后台已落地的表格、筛选表单、详情页、状态 badge 模式
- 颜色、聚焦态、键盘可达性等可访问性说明
