# 部署与构建参考

## 文档目标

整理当前仓库与部署、构建、打包最相关的项目化结论，供后续平台接入或 CI/CD 设计复用。

## 当前仓库口径

- Monorepo：`pnpm workspace` + `turbo`
- 后端：`apps/api`
- 前端交付面：`apps/admin`、`apps/miniapp` H5 构建产物
- 当前默认本地运行方式是“基础设施 Docker + 应用宿主机”

## 本地与联调方式

### 默认模式

- 入口：`docker compose up -d mysql redis`，然后宿主机执行 `pnpm dev:api`、`pnpm dev:admin`、`pnpm dev:miniapp:h5`
- 适用：日常本地开发、接口调试、前后端联调
- Redis 默认映射到宿主机 `6380`，避免与开发机已安装的本地 `6379` 实例冲突

### 热更新模式

- 入口：`docker compose -f docker-compose.yml -f docker-compose.hot.yml up --build mysql redis api admin-dev miniapp-h5-dev`
- 适用：需要把 API、后台、小程序 H5 一并放进容器热更新的场景

### 全栈模式

- 入口：`docker compose -f docker-compose.yml -f docker-compose.full.yml up -d --build`
- 适用：完整联调、演示、验收前预检

## 与部署直接相关的结论

- 默认应从仓库根目录安装依赖，而不是假设每个 app 都能完全独立安装
- 构建优先复用根脚本和 `turbo` 任务，而不是临时拼装命令
- Prisma schema 或 lockfile 变更后，应显式重建 API 相关镜像或依赖环境
- 当前仓库还没有正式收口的云端部署方案；`Nixpacks` 等仅作为后续候选

## 后续平台接入时最先确认的事项

- 实际独立部署目标是 `apps/api`、静态前端，还是整个 Monorepo
- 环境变量来源与注入方式
- Prisma migrate / db push / seed 在部署链路中的位置
- 构建缓存、依赖缓存和镜像分层策略
