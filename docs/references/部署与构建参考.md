# 部署与构建参考

## 文档目标

整理当前仓库与部署、构建、打包最相关的项目化结论，供后续平台接入或 CI/CD 设计复用。

## 当前仓库口径

- Monorepo：`pnpm workspace` + `turbo`
- 后端：`apps/api`
- 前端交付面：`apps/admin`、`apps/miniapp` H5 构建产物
- 当前最稳定的本地运行方式仍是 Docker Compose

## 本地与联调方式

### 全栈模式

- 入口：`docker compose up -d --build`
- 适用：完整联调、演示、验收前预检

### 热更新模式

- 入口：`docker compose -f docker-compose.yml -f docker-compose.hot.yml up --build mysql api admin-dev miniapp-h5-dev`
- 适用：API、后台、小程序 H5 联调开发

### 宿主机模式

- 入口：仅启动 MySQL，其他服务走宿主机 `pnpm`
- 适用：本地纯开发或快速调试

## 与部署直接相关的结论

- 默认应从仓库根目录安装依赖，而不是假设每个 app 都能完全独立安装
- 构建优先复用根脚本和 `turbo` 任务，而不是临时拼装命令
- Prisma schema 或 lockfile 变更后，应显式重建 API 相关镜像或依赖环境
- 当前仓库还没有正式收口的云端部署方案；`Nixpacks` 等仅作为后续候选

## 后续平台接入时最先确认的事项

- 实际独立部署目标是 `apps/api`、静态前端，还是整个 Monorepo
- 环境变量来源与注入方式
- Prisma migrate / db push / seed 在部署链路中的位置
- 构建缓存、依赖缓存和镜像分层策略
