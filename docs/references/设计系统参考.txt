面向 LLM 的设计系统参考笔记

适用场景
- 设计或实现新的后台页面
- 判断某个页面是否偏离当前后台视觉和交互模式

本仓库结论
- 后台管理端基于 React + Vite + shadcn/ui，并以 `shadcn-admin` 作为 UI 基座。
- 新页面应像“在现有后台产品上扩展出来”，而不是独立演示页。
- 共享 UI 组件只在多个页面模式稳定后再抽离。

页面模式
- 列表页默认采用“页头 + 筛选卡片 + 数据表格卡片”。
- 页面头部通常是“模块路径 + 页面标题 + 一句话说明 + 右上角主操作”。
- 状态字段优先用 `Badge`；时间字段优先格式化为本地可读时间。
- 高风险动作优先通过确认弹窗承接。

状态与字段展示
- `Badge.default`：启用、正常、已完成、已公开、已通过。
- `Badge.secondary`：待处理、未发放、待领取、冻结、停用、待接收。
- `Badge.destructive`：已作废、已驳回、已下架、高风险异常。
- 列表列顺序优先遵循“识别字段 -> 主信息 -> 关系字段 -> 状态 -> 时间 -> 长文本”。

不要做的事
- 不要把第三方展示页当成产品级信息架构。
- 不要引入与当前后台基座无关的视觉母题。
- 不要由前端自行发明业务状态含义。

后续待确认
- 后台统一间距、字体、色板和可访问性细则
- 已落地页面模式是否需要继续抽成共享组件
