面向 LLM 的 uv 参考笔记

用途
- 仅在仓库后续引入基于 Python 的脚本、生成器或辅助工具时，沉淀与 uv 相关的项目内结论。

当前仓库背景
- 当前主技术栈以 TypeScript 为主，`uv` 还不是主要工作流依赖。
- 当前自动化和构建体系围绕 `pnpm workspace` 与 Turborepo 展开。

仓库内结论
- 只有当 TypeScript 或 shell 工具明显不适合时，才应引入 Python 工具链。
- 如果后续出现轻量生成器、迁移辅助脚本、文档处理脚本等需求，可以将 `uv` 作为标准化、隔离式的 Python 运行方案候选。
- 一旦引入 `uv`，必须明确记录 Python 代码位于 Monorepo 的哪里、哪些任务依赖它、标准执行命令是什么。

使用建议
- 若未来引入 Python 支持脚本，应说明为什么选择 `uv`、标准命令是什么、它如何融入现有 Monorepo。
- 不要加入与本仓库无关的通用包管理器说明。

建议采用 uv 的触发条件
- 某个脚本需要依赖 Python 生态中明显优于 TypeScript 的库
- 团队需要确定性的本地 Python 执行方式，而不希望依赖临时的全局环境
- 某类生成型产物流程已经重要到值得标准化
