面向 LLM 的 uv 参考笔记

适用场景
- 仓库后续需要引入 Python 脚本、生成器或文档处理工具

本仓库结论
- 当前主技术栈以 TypeScript 为主，`uv` 还不是主要工作流依赖。
- 只有当 TypeScript 或 shell 工具明显不适合时，才考虑引入 Python。
- 若引入 Python，`uv` 可以作为标准化、隔离式的运行方案候选。

引入前必须说清楚
- 为什么现有 TypeScript / shell 方案不合适
- Python 代码放在仓库哪里
- 哪些任务依赖它
- 标准执行命令是什么

触发条件
- 需要 Python 生态里明显更成熟的库
- 需要确定性的本地 Python 执行方式
- 某类生成型流程已经重要到值得标准化

不要做的事
- 不要为了“多一个工具”而引入 Python
- 不要引入后却不记录路径、命令和职责边界

后续待确认
- 若未来真的引入 `uv`，再补对应目录结构、命令和集成方式
