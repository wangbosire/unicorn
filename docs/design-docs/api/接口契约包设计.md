# 数字藏品运营与展示平台 `api-contracts` 包设计

## 文档目标

说明 `packages/api-contracts` 为什么存在、怎么分目录、什么时候应该往里放内容。

## 真相源

- 包目录：`packages/api-contracts/src/**`
- 使用方：`apps/api`、`apps/admin`、`apps/miniapp`

## 包职责

- 存放前后端共享 DTO、query、params、response 类型
- 统一导出接口契约，避免前后端各写一份
- 不放业务实现、数据库模型、页面状态

## 与其他共享包的边界

- `@unicorn/api-contracts`：接口输入输出
- `@unicorn/shared-types`：领域类型、枚举、共享业务结构
- `@unicorn/shared-config`：权限 key、事件名、路由常量等

## 目录规则

推荐按接口边界分层：

```text
src/
  common/
  admin/
  member/
  public/
```

资源再按模块拆分：

```text
src/admin/series/
src/admin/issuance-batches/
src/member/my-collections/
src/public/collections/
```

## 文件规则

- 一个文件只放一种契约对象
- 文件名直接表达用途：
  - `*.request.ts`
  - `*.response.ts`
  - `*.query.ts`
  - `*.params.ts`
- 每个资源目录保留 `index.ts`

## 何时新增文件

- 新接口需要新的请求体、响应体、query 或 params
- 同一资源下已有文件难以保持单一职责
- 某结构被前后端重复引用

## 维护规则

- 先改 `api-contracts`，再改前后端调用方
- 契约变化时同步更新资源目录下的 `index.ts`
- 不在本文维护完整文件树，目录级真相以 `packages/api-contracts/src/**` 为准
