# 数字藏品运营与展示平台代码注释与字段说明规范

## 文档目标

让人和 AI 都能快速看懂“哪些地方必须写说明、说明至少写到什么程度”。

## 真相源

- 共享契约：`packages/api-contracts/src/**`
- 共享类型：`packages/shared-types/src/**`
- Prisma：`apps/api/prisma/schema.prisma`
- 端侧 API 层：`apps/*/src/apis/**`

## 总规则

- 注释默认写“业务含义”和“约束原因”，不写废话
- 能靠类型名看懂的，不补重复注释
- 一段逻辑如果离开业务上下文就难懂，必须补说明
- 共享类型、DTO、Prisma 模型、状态字段优先补说明

## 必须写说明的地方

### 共享类型与接口契约

- `packages/api-contracts`
- `packages/shared-types`
- 规则：字段要能看出业务含义、单位、可选条件、状态约束

### Prisma 模型

- 核心模型要说明“这张表为什么存在”
- 状态字段、JSON 字段、业务编号字段要说明用途

### 后端 DTO / Service / Policy

- DTO：补字段语义、校验意图、状态限制
- Service：补复杂流程、事务边界、幂等点
- Policy / Guard：补权限判断依据

### 前端 API 层与页面状态模型

- `src/apis/**`：说明字段映射、错误壳解包、端侧兼容口径
- 页面状态：说明状态来源和与后端枚举的对应关系

## 字段说明最低要求

- 这是什么
- 谁会写入或消费
- 有什么约束
- 状态字段取值代表什么

## 推荐写法

### 类型字段

```ts
/**
 * 会员当前可操作的藏品数量。
 * 仅统计状态为 OWNED 的藏品，不包含已转让完成记录。
 */
ownedCollectionCount: number;
```

### Prisma 模型

```prisma
/// 藏品内容版本快照；每次提交审核都应落一版，避免覆盖历史内容。
model CollectionContentVersion {
  /// 当前版本的编辑态，和发布态、审核态分开维护。
  editStatus String
}
```

### 服务逻辑

```ts
// 同步机审占位与发布态写入必须放在同一事务里，避免出现“已公开但无审核记录”的脏状态。
```

## 不建议写的注释

- “给变量赋值”
- “调用某个函数”
- 与代码字面含义完全重复
- 已经过期但没人维护的历史说明

## 特别关注字段

- 时间字段：时区、格式、是否可空
- JSON 字段：结构来源、允许为空时的语义
- 状态字段：每个值的业务意义
- 错误码字段：由谁消费、是否面向前端展示

## AI 友好要求

- 一条注释只解释一个判断
- 优先短句，不写大段背景故事
- 若已有专题文档，注释里只写必要结论，不复制文档正文

## 执行清单

- 新增共享类型时补字段说明
- 新增 Prisma 模型时补模型说明与关键字段说明
- 新增复杂事务时补“为什么这样做”
- 新增状态字段时补各状态含义
