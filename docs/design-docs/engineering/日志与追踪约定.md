# 日志与追踪约定

## 文档目标

明确 `apps/api` 的日志格式、级别、写入点和追踪标识，保证关键业务行为可被检索、串联与告警。

## 总规则

- 全部日志输出为单行 JSON（structured logs），便于聚合系统直接消费。
- 每条日志携带 `requestId`，HTTP 入口和 BullMQ Worker 都要可贯通。
- 不打敏感字段：手机号、激活码原值、JWT、密码哈希、`Authorization` header。
- 不在循环里逐条打日志；聚合后输出一条带计数/汇总的日志即可。
- 日志失败不影响主流程。

## 字段集

每条日志至少包含：

| 字段 | 类型 | 说明 |
|---|---|---|
| `ts` | string | ISO8601 时间戳 |
| `level` | enum | `debug` / `info` / `warn` / `error` |
| `ctx` | string | 输出来源，通常是类名（如 `CollectionActivationService`） |
| `requestId` | string \| null | 请求级唯一标识；Worker 上下文为 null（用 `jobId` 替代） |
| `msg` | string | 人类可读的简短消息 |

可选稳定字段：

| 字段 | 说明 |
|---|---|
| `event` | 稳定事件名，例如 `collection.activated`、`transfer.cancelled`；用于告警和报表 |
| `jobId` | BullMQ Worker 上下文中的任务 ID |
| `durationMs` | 操作耗时；包裹在关键动作两侧 |
| `code` | `BizError` 的业务错误码 |
| `actor` | 触发动作的主体 ID（会员、后台用户） |
| 业务字段 | `collectionId`、`memberId`、`transferId`、`fromStatus`、`toStatus` 等平铺，不要嵌套对象 |

## 级别使用

- `debug`：详细的开发追踪；生产默认关闭。
- `info`：状态变更、关键动作完成、副作用入队成功。**这是默认级别。**
- `warn`：业务拒绝（`BizError`）抛出前；预期内的失败（鉴权失败、过期等）。
- `error`：未处理异常、外部依赖失败、worker 重试耗尽。

## 写入点

四类必写的日志事件：

1. **状态变更**：实体跨越状态机时
   - `event=collection.activated`，包含 `collectionId`、`fromStatus`、`toStatus`、`actor`
2. **关键动作完成**：激活、审核通过/驳回、转让创建/接收/撤销、通知派发等
   - 包含动作名、关键 ID、`durationMs`
3. **副作用触发**：通知入队、Worker 执行、外部调用
   - 包含 `channel`、`messageId`、`jobId` 等
4. **业务拒绝**：在 `throw new BizError(...)` 之前 `warn` 一条
   - 包含 `code` 与导致拒绝的关键上下文字段

不必写日志的场景：

- 纯查询、列表读取（除非耗时异常）
- 已被 `ApiExceptionFilter` 自动捕获的 5xx — 不要重复打
- DTO 校验失败 — 也由过滤器统一处理

## 敏感字段黑名单

下列字段**禁止**出现在日志中（即使被 `payload` 间接携带）：

- `password`、`passwordHash`、`pwd`
- `activationCode`（原值）、`transferCode`（原值）
- `authorization`、`token`、`accessToken`、`refreshToken`
- `mobile`（手机号原值；如必要可记 hash 或脱敏 `138****1234`）
- 任何 `*.secret`、`*.key`

## 请求追踪

- 入口中间件从 `x-request-id` header 读取；缺失则生成 UUID v4。
- ID 存入 `AsyncLocalStorage`，本次请求生命周期内任何 service 调用 `Logger.log()` 时会自动带上。
- HTTP 响应自动回写同名 header，便于前后端联调和客诉定位。
- 长度限制：来源 header 截断到 64 字符，避免日志膨胀。

## Worker 上下文

- BullMQ Worker 不存在 HTTP 请求，没有 `requestId`。
- Worker 处理任务时改用 `jobId` 作为追踪标识；如可获取原始 dispatch 时的 `requestId`，作为业务字段携带（`originRequestId`）。
- Worker 自己的进入/退出/失败日志按本约定执行，`event` 前缀使用 `notification.dispatch.*`。

## AI 阅读建议

- 想看格式与字段：本文「字段集」。
- 想看在哪里加日志：本文「写入点」。
- 想看追踪如何贯通：本文「请求追踪」与「Worker 上下文」。

## 维护规则

- 新增稳定事件名（`event`）前先在本文登记，避免散落。
- 修改字段集要同步刷新真相源中的 Logger 实现。
- 不在本文记录具体实现细节，实现细节以代码为准。
