# 数字藏品运营与展示平台后端测试策略

## 文档目标

只回答三个问题：测试放哪、优先测什么、什么时候需要上 HTTP 集成测试。

## 当前结论

- 测试框架：`Vitest`
- HTTP 集成测试：`supertest`
- 测试根目录：`apps/api/test`
- 断言优先可读、直接，业务错误优先断言错误码

## 目录规则

- `test/` 尽量镜像 `src/`
- 公共层放 `test/common/**`
- 业务模块放 `test/modules/**`

示例：

- `src/modules/issuance/series/series.service.ts`
- `test/modules/issuance/series/series.service.test.ts`

## 分层规则

### 单元测试

- 测公共工具、响应壳、异常转换、序列化
- 不启动 HTTP

### Service 测试

- 优先覆盖业务规则、状态流转、错误码、事务边界
- 直接 mock Prisma 或外部依赖
- 是新增业务能力时的第一优先级

### Controller 测试

- 只测参数透传、headers 组装、调用边界
- 不重复覆盖 service 业务规则

### HTTP 集成测试

- 只给主链路和高风险边界能力上样板覆盖
- 重点验证路由、全局拦截器、统一响应壳、全局异常过滤器

## 优先覆盖范围

- 状态流转
- 业务错误码
- 分页结构
- 时间字段序列化
- 权限或归属校验

## 当前主链路样板

- `admin-api/series`
- `admin-api/issuance-batches`
- `admin-api/activation-codes`
- `member-api/auth/me`
- `member-api/my/collections`
- `member-api/collection-activation`

## 新增测试顺序

1. 先补 service 测试
2. 再补 controller 测试
3. 最后只给关键路径补 HTTP 集成测试
