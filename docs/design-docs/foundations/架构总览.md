# 架构总览

## 文档目标

给 AI 和研发一个足够短的系统地图，回答“仓库怎么分、职责怎么切、什么规则不能破”。

## 真相源

- 后端实现：`apps/api/src/modules/**`
- Prisma：`apps/api/prisma/schema.prisma`
- 契约：`packages/api-contracts/src/**`
- 共享类型：`packages/shared-types/src/**`

## 一句话架构

这是一个以 `apps/api` 为业务真相源、由 `apps/admin` 和 `apps/miniapp` 分别承载后台与会员体验的 Monorepo。

## 技术基线

- Monorepo：`pnpm workspace` + `turbo`
- 后台前端：React + Vite + `shadcn-admin`
- 小程序前端：TaroJS
- 后端：NestJS + Prisma
- 共享层：`packages/*`

## 仓库结构

- `apps/admin`：后台管理端
- `apps/miniapp`：会员端小程序
- `apps/api`：唯一后端服务
- `packages/api-contracts`：接口契约真相源
- `packages/shared-types`：共享类型与枚举
- `packages/shared-config`：共享常量、权限 key、事件名
- `docs/`：产品、设计、计划、归档资料

## 应用边界

### `apps/admin`

- 面向运营、审核、客服、系统管理员
- 负责系列、批次、激活码、审核、会员治理、通知、转让、仪表盘
- 不承载会员身份逻辑

### `apps/miniapp`

- 面向会员
- 负责登录、激活藏品、我的藏品、内容编辑、消息、转让、公开页跳转
- 不承载后台 RBAC

### `apps/api`

- 对外提供 `admin-api`、`member-api`、`public-api`
- 负责状态流转、权限判断、持久化、审计与事件编排
- 是业务规则、状态规则、权限规则的最终收口点

## 后端业务域

- `iam`：后台账户、角色、权限
- `member`：会员主账号、微信绑定、会员状态
- `issuance`：系列、批次、激活码、发行链路
- `collection`：藏品主实体、归属、内容版本、公开展示
- `review`：机审、人工复核、审核历史、下架
- `comment`：评论、回复、评论审核
- `transfer`：转让单、接收、流转记录
- `notification`：消息、派发记录、通知摘要
- `audit`：操作日志与追溯

## 不能破的架构约束

### 身份与权限

- 后台用户与会员账户严格隔离
- 后台权限使用 RBAC
- 会员权限使用“登录态 + 资源归属 + 状态”判断
- 公开接口不得泄露内部审核与治理信息

### API 边界

- `admin-api` 只服务后台
- `member-api` 只服务会员登录态
- `public-api` 只服务公开展示与匿名读取

### 共享层

- DTO 真相源在 `packages/api-contracts`
- 枚举与共享类型优先沉淀到 `packages/shared-types`
- 权限 key、事件名、跨端常量优先沉淀到 `packages/shared-config`
- 前端页面内不要各自发明一套接口字段

### 状态与持久化

- 状态机先于页面行为
- 审核态、发布态、编辑态分开建模
- 关键历史记录默认可追溯，不做覆盖式丢失
