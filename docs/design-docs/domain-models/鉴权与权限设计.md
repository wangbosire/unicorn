# 数字藏品运营与展示平台鉴权与权限设计

## 文档目标

只保留身份边界、权限判断方式和不能破的安全规则。

## 三类身份

- 后台用户：运营、审核、客服、系统管理员
- 会员用户：小程序或公众号登录会员
- 匿名访客：公开页浏览者

## 核心规则

### 后台用户

- 独立登录入口
- 独立账户表
- 使用 RBAC

### 会员用户

- 通过微信授权接入
- 统一归并到会员主账户
- 不使用后台 RBAC
- 通过“登录态 + 资源归属 + 资源状态”授权

### 匿名访客

- 无需登录
- 只能访问公开内容
- 不可看到审核、风控、隐私信息

## 后台权限模型

- 权限点：最小授权单元，语义固定在 `action`
- 权限组：一组相关 `action` 的聚合，用于菜单、页面域、批量授权与阅读理解

建议口径：

- `Permission` 表只表达一个稳定 `action`，例如 `members.read`、`members.freeze`、`roles.assign_permissions`
- 新增 `PermissionGroup` 表达一组 action，例如 `members`、`notifications`、`system.roles`
- 菜单、页面域、批量授权优先面向 `PermissionGroup`
- 运行时鉴权仍以 `Permission` 为准，不直接用 `PermissionGroup` 判定放行

当前实现口径：

- 后端以 `permissionKeys` 写入后台 JWT，并由 Guard 在接口层统一校验。
- 前端导航由菜单配置接口返回，运行时按权限组与 action 命中结果过滤菜单可见性。
- 当前已稳定落地：业务域权限、系统管理权限、菜单数据化管理，以及页面 `403` / 按钮隐藏 / 已登录会话失效收敛。

推荐实现方式：

- 角色授权 UI 可以按“权限组”勾选
- 落库时把组内 action 展开写入 `RolePermission`
- Guard / Controller / Service 只认 action 级 `permissionKey`

这样可以避免两层授权真相源并存：  
`PermissionGroup` 负责组织与配置，`RolePermission` 仍是最终授权真相源

典型动作权限：

- `issuance.series.create`
- `issuance.activation-code.generate`
- `collection.review.approve`
- `collection.review.reject`
- `comment.review.approve`
- `member.freeze`

## 当前未收口项

### 菜单与权限层级未正式分层

- `collections.manage`、`collection_reviews.manage`、`collection_comments.manage` 仍是“页面入口 + 动作能力”混合命名。
- `members.manage` 仍未继续细化到冻结 / 解冻等更小 action。
- 评论治理与藏品治理尚未像发行、通知、转让一样完全拆成“read + action”分层。

## 建议的数据模型

建议将当前模型从“角色 - 权限点”扩展为“四层”：

1. `Permission`
   - 原子 action
   - 例：`members.read`、`members.freeze`、`roles.read`、`roles.assign_permissions`
2. `PermissionGroup`
   - action 聚合
   - 例：`members`、`collection_comments`、`system.roles`
3. `PermissionGroupItem`
   - group 与 action 的映射关系
4. `RolePermission`
   - 角色最终拥有的 action 权限

可选补充：

- `Menu`
  - 菜单主数据独立建表，不继续把菜单结构硬编码在前端配置里
- `MenuPermissionGroup`
  - 菜单与权限组通过关联表绑定，一个菜单可命中多个权限组，语义为 `anyOf`
- `AuthorizationChangeLog`
  - 若后续需要审计“通过哪个权限组授予了哪些 action”或“菜单绑定发生了什么变化”，统一落到权限变更日志

建议字段约束：

- `Permission.permissionKey`
  - 稳定代码主键，一旦投入使用不得随意修改
- `Permission.isBuiltin`
  - 标记内置权限；内置权限允许停用，不允许删除或改 key
- `PermissionGroup.groupKey`
  - 稳定分组主键，供菜单、页面域和配置界面引用
- `MenuPermissionGroup`
  - 只负责菜单进入条件，不直接参与接口放行

## 建议的命名分层

建议把 action 权限收敛成稳定模式：

- `*.read`：列表、详情、页面进入
- `*.create`
- `*.update`
- `*.delete`
- `*.review`
- `*.approve`
- `*.reject`
- `*.export`
- `*.assign_roles`
- `*.assign_permissions`
- `*.enable`
- `*.disable`

说明：

- `manage` 可作为过渡阶段兼容 key，但不建议成为长期主模型
- 长期看，`manage` 应拆成更细 action，避免一个 key 覆盖过多风险动作

## 补充约束

### 稳定主键规则

- `permissionKey` 是代码与数据库共享的稳定主键，不把“改名”当配置能力开放
- 可配置项应限定在：显示名称、描述、权限组、排序、状态、菜单绑定、角色授权
- 一旦某个 action 已被控制器、前端或脚本引用，修改 key 视同破坏性变更

### 内置权限与自定义权限

- 内置权限：系统主链路依赖的 action，必须保留 `isBuiltin=true`
- 内置权限可停用、可隐藏，但不允许删除、不允许改 key
- 自定义权限：仅在未来确有可扩展场景时开放；若当前没有真实需求，可以先不开放创建入口

### 超级管理员语义

- 继续保留 `*` 作为超级管理员全权标记
- `*` 在运行时仍由 Guard / 前端权限判断逻辑直接特判放行
- 即使角色授权 UI 按权限组与 action 展示，超级管理员角色仍可通过 `*` 表达“全量拥有”
- 为避免 UI 与运行时语义割裂，角色详情页应同时展示“超级管理员（*）”与当前系统可识别的权限范围摘要

### 内置角色与自定义角色

- 角色区分 `isBuiltin`
- 内置角色：系统默认治理角色，可改显示名称、描述、排序、启停状态
- 内置角色不允许删除，不允许改稳定 `roleKey`
- 自定义角色：由后台按需创建，用于补充组织分工
- 若一个内置角色已被主链路或种子逻辑依赖，应限制其关键权限被整体清空

### 真相源规则

- `PermissionGroup` 只负责组织、展示、批量勾选和菜单绑定
- `RolePermission` 才是最终授权真相源
- 角色在 UI 上勾选权限组时，保存阶段必须把组内 action 展开后写入 `RolePermission`
- Guard / Controller / Service 不直接按权限组放行

### 权限组结构规则

- 权限组只允许单层结构，不支持组嵌套组
- 禁止出现父组 / 子组继承关系，避免形成隐式授权链
- 一个 action 可归属多个组，但应尽量控制在少数明确场景，避免权限语义重复扩散

### 审计规则

- 角色与权限的授予、撤销、启停都必须留审计日志
- 至少记录：操作人、目标角色或后台用户、变更前、变更后、变更来源、时间
- 若一个权限组触发了多条 action 授权，日志里应能还原“通过哪个组产生了哪些授权”

### JWT 与缓存失效

- 当前后台 JWT 内含 `permissionKeys`，角色变更后老 token 不会自动失效
- 至少需要一种补偿策略：
  - 缩短后台 token 生命周期
  - 或引入 `permissionVersion` / `roleVersion`，鉴权时校验版本
  - 或在高风险后台场景强制重新登录
- 产品口径：保存后对新登录立即生效；已登录用户最长 `5` 分钟内生效
- 在没有版本校验前，不应把“改完角色立即全站生效”当成保证

### 前端无权限行为

- 菜单：无权限时直接隐藏
- 直链进入页面：返回或展示 `403`
- 页面内按钮：无权限时隐藏，不采用仅禁用但仍展示的策略
- 前端隐藏只用于体验收口，不替代后端 action 鉴权

### 菜单与权限组绑定

- 菜单优先绑定 `PermissionGroup`
- 一个菜单可配置多个权限组，语义为 `anyOf`
- 命中任一权限组即可展示菜单
- 菜单不直接表达 `allOf`，避免配置复杂度过高

### 孤儿 action 与失效菜单

- 孤儿 action（未归属任何权限组）不自动出现在配置界面主列表或菜单体系中
- 孤儿 action 仍可作为底层授权对象存在，但应被运营配置界面标记为异常数据
- 菜单若绑定了失效或不存在的权限组，应自动隐藏
- 这类失效绑定需要产生告警或配置校验提示，避免“菜单静默消失但无人感知”

### 迁移与兼容

- 现有 `*.manage`、`nav.m2_placeholder` 属于过渡口径，不应长期保留
- 新模型落地时需要明确：
  - 旧 key 如何映射到新的 action 集
  - 哪些菜单先切到 `PermissionGroup`
  - 哪些控制器继续兼容旧 key，兼容多久
- 迁移期允许“旧 key + 新 action”并存，但必须有明确下线时间

- 最终落库：仍写 action 级 `RolePermission`

## 会员权限模型

会员侧不建角色树，优先判断：

- 是否已登录
- 是否为资源拥有者
- 当前状态是否允许操作

典型例子：

- 只有当前拥有者可编辑藏品
- 冻结会员不可继续评论或转让
- 转让中藏品不可再次发起转让

## 公开访问规则

公开接口只返回：

- 已公开内容
- 对外可展示评论
- 不含内部审核信息的展示字段

## 接口落点

- `admin-api`：后台身份 + RBAC
- `member-api`：会员身份 + 归属校验 + 状态校验
- `public-api`：匿名可访问，仅暴露公开数据

## 一期必须落实

- 后台用户与会员不得混用账户体系
- 所有审核动作必须记录操作人
- 所有后台修改类接口必须做动作级鉴权
- 会员侧编辑、评论、转让必须做归属校验
