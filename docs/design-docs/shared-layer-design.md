# 共享层设计

本文档将 Monorepo 中共享层的职责边界从架构总览中单独抽出，便于包设计、职责划分和抽取顺序讨论时快速引用。

## 设计目标

- 为多端与后端提供稳定的共享类型、契约、配置与工具层
- 减少重复定义状态、接口、常量和基础样式的成本
- 明确哪些能力应该抽到共享层，哪些能力应保留在应用层

## 包职责

### `api-contracts`

用于共享前后端接口约定，建议按 API 边界组织：

- `admin`
- `member`
- `public`

### `shared-types`

用于共享：

- 状态枚举
- DTO 类型
- 视图模型
- 通用响应类型

### `shared-config`

用于共享：

- 权限 key
- 路由常量
- 事件名
- 通用常量

### `ui`

用于承载：

- 后台与小程序可复用的设计 token
- 通用样式约定
- 若技术方案兼容，可抽取少量跨端组件

### `utils`

用于承载纯工具逻辑，不承载业务规则。

## 抽取原则

- 共享层只承接稳定、跨模块复用、语义明确的内容
- 业务规则优先留在后端业务模块，不因为复用诉求过早抽象
- 状态枚举、接口契约、权限 key 这类高频基石应优先沉淀到共享层
- 不要把应用层暂时性的页面结构、业务流程编排和局部实现细节强行抽成共享包
