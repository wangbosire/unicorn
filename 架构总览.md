# 架构总览

本文档用于描述仓库的顶层架构地图、应用边界、共享层职责和研发推进原则，作为 `docs/design-docs/` 中详细设计文档的总入口。

## 系统目标

本仓库承载“数字藏品运营与展示平台”一期建设，目标是围绕以下主链路完成产品闭环：

- 后台发行数字藏品
- 通过激活码向会员发放藏品
- 会员编辑藏品展示内容并提交审核
- 审核通过后对外公开展示
- 支持评论、转让、通知和后台运营治理

## 顶层应用地图

### `apps/admin`

后台管理端，面向运营、审核、客服和系统管理员，负责：

- 系列、批次、激活码、藏品等运营管理
- 内容审核、评论审核、转让记录处理
- 会员查询、冻结解冻、日志与权限管理

### `apps/miniapp`

会员端微信小程序，面向 C 端会员，负责：

- 微信登录与会员身份接入
- 激活藏品、查看我的藏品、编辑内容
- 查看公开展示、消息、转让和个人中心

### `apps/api`

统一后端服务，采用 NestJS 模块化单体，负责：

- 后台、会员、公开访问三类 API 出口
- 业务领域模型、状态流转、审核编排
- 鉴权、持久化、事件、审计和外部平台接入

## 仓库分层

### 应用层

- `apps/admin`：后台前端
- `apps/miniapp`：会员前端
- `apps/api`：后端服务

### 共享层

- `packages/api-contracts`：前后端共享接口约定
- `packages/shared-types`：领域类型、状态枚举、DTO 类型
- `packages/shared-config`：权限 key、事件名、常量配置
- `packages/ui`：前端共享 UI 组件
- `packages/utils`：纯工具函数
- `packages/eslint-config`：统一 lint 规则
- `packages/tsconfig`：统一 TS 配置

### 文档与辅助层

- `docs/product-specs`：产品规格与业务规则
- `docs/design-docs`：架构、模型、状态机、接口等设计文档
- `docs/exec-plans`：执行计划、里程碑与实施顺序
- `docs/references`：外部参考资料
- `scripts`：初始化、批处理、测试数据辅助脚本
- `infra`：部署、环境和基础设施配置

## API 边界

后端对外固定为三类接口边界：

- `admin-api`：后台管理端接口
- `member-api`：会员端接口
- `public-api`：公开展示与匿名可访问接口

边界原则：

- 后台用户与会员账户严格隔离
- 后台权限使用 RBAC
- 会员权限使用资源归属和状态校验
- 公开接口不暴露内部审核与风控信息

## 核心业务域

一期建议围绕以下业务域组织后端模块：

- `iam`：后台账户、角色、菜单、权限
- `member`：会员主账号、微信绑定、会员状态
- `issuance`：系列、批次、激活码、发放记录
- `collection`：藏品主实体、归属关系、内容版本、公开页
- `review`：内容机审、人工复核、审核记录
- `comment`：评论、回复、评论审核
- `transfer`：指定会员转让、转让码转让、转让记录
- `notification`：站内信、站外通知、模板与发送记录
- `audit`：操作日志、业务审计与追溯

## 当前落地状态

当前仓库的一期主链路与运营治理能力已完成收口，状态如下：

- `admin` 已接入仪表盘、通知中心、转让记录、审核治理、会员治理等后台主链路页面
- `miniapp` 已接入登录、激活、我的藏品、内容编辑、公开展示、消息与基础转让能力
- `api` 已形成覆盖 admin-api、member-api、public-api 的一期业务闭环与验收基线
- `docs` 已形成 PRD、架构设计、执行计划、验收清单与专项归档资料

当前阶段的重点不是继续堆叠新页面，而是收口稳定结论、整理归档资料，并为下一阶段规划保留清晰基线。

## 研发推进原则

- 优先完成“发码 -> 领藏品 -> 编辑内容 -> 审核 -> 展示”闭环
- 先稳定主模型和状态模型，再扩展评论、转让、通知
- 页面结构可以先占位，但业务规则必须由后端统一收口
- 共享枚举、契约、常量优先沉淀到 `packages/*`
- 所有稳定结论优先写入 `docs/`，避免口头约定失真
